Installation

Aithernet is invitation-only. Packages are delivered through the authenticated portal after your access request is approved — this public site hosts no binaries, keys or credentials.

The customer sequence

  1. Request early access with the early-access request form.
  2. Get your invitation — an early-access invitation email is sent to the address you submit (during controlled early access this may be reviewed first).
  3. Create your account from the invitation link and sign in to the portal.
  4. Download the signed release (Debian package, plus wheel/sdist) from the portal's Downloads page.
  5. Verify the release: check the detached signature over the manifest with the published verification key, then verify each artifact checksum.
  6. Install the package and run setup.
This page is a high-level summary. After you sign in and your release is authorized, the portal's Getting started page holds the complete, copyable post-login journey — verify & install, standalone software-only setup, Gemini API / Codex provider configuration, the workstation service, node verification and running a software-only mission. sudo is for APT only; never use pip for GNU Radio, SoapySDR, UHD, libiio or python3-venv.
The current beta is a one-command experience: after aithernet setup, run your work with a single command — aithernet run "<what you want done>" — which creates, starts, monitors and returns the result. No separate submit/start/watch, no editing systemd units, PATH or YAML, and no knowing provider internals. If anything drifts, aithernet doctor --repair fixes the common cases. Physical SDR transmission is available when you enable and configure it locally on TX-capable hardware (you select the parameters and operate within your local rules).

Verify & install (Ubuntu 24.04 amd64)

From the portal Downloads page, download (into one directory) the package plus the verification files served alongside it: manifest.json, manifest.sig, the production public key aithernet-release.pub (PEM), and SHA256SUMS. Verify before installing, using only standard tools:

# 1) BEFORE installing — verify checksums, then the detached Ed25519 signature
#    over the manifest with the production public key (standard OpenSSL, no Aithernet tools yet)
sha256sum -c SHA256SUMS
openssl pkeyutl -verify -pubin -inkey aithernet-release.pub \
  -rawin -in manifest.json -sigfile manifest.sig

# 2) refresh APT package indexes FIRST, then install the node package.
#    `apt update` is required on a fresh Ubuntu image — without it APT has no candidate for
#    python3-venv (and the recommended gnuradio / uhd-host / soapysdr-tools), and the install
#    fails with "python3-venv but it is not installable".
sudo apt update
sudo apt install ./aithernet_1.0.0~beta.3_amd64.deb

# 3) the documented CLI is now on PATH
aithernet --help

# 4) AFTER installing — an optional all-in-one re-check of the same downloaded directory
#    (this command ships in the .deb, so it is only available once installed)
aithernet release verify ./
APT resolves the package's dependencies (python3-venv) and the recommended SDR packages (gnuradio, uhd-host, soapysdr-tools) from Ubuntu's signed main/universe repositories. If apt install reports a package "is not installable" on a fresh machine, run sudo apt update and retry. Do not use pip for GNU Radio, SoapySDR, UHD or python3-venv — they are system packages. The bundled installer can do this for you: ./aithernet-install.sh --deb ./aithernet_1.0.0~beta.3_amd64.deb (validates the platform, refreshes stale indexes, retries, never uses pip).

Set up and run (1.0.0-beta.3)

# one guided setup, then run your work with one command
aithernet setup
aithernet run "<what you want Aithernet to do>"
  • aithernet setup detects and configures your local workstation: it initializes the canonical node identity and state, guides you through connecting a coordinator (planner) and a coding agent in plain language, configures and verifies the managed user service, resolves supported CLI runtimes automatically, verifies GNU Radio and the managed signed RF-MCP component, and runs a software-only readiness check.
  • aithernet run "<prompt>" creates the mission, starts it, shows progress, returns the result, and prints the mission, run and artifact IDs — one command, no separate submit/start/watch.
  • aithernet doctor --repair safely repairs common local configuration drift.
  • No manual systemd units, YAML, PATH/NVM, environment files, or provider-adapter knowledge is expected. upgrading from an earlier beta? Just install the new package and run aithernet setup — it detects and migrates the existing install (preserving your node ID, name, missions, events, artifacts and provider configuration).

Signed-in customers get the exact versioned commands, the authorized filenames, and the verification key on the portal Getting started and Downloads pages.

Advanced / operator workflows (not needed for the standard journey)

These remain available for operators and automation, but are not part of normal onboarding. The standard journey above is setup → run.

  • Connect providers explicitly: aithernet agents connect coordinator / aithernet agents connect coding.
  • Lower-level mission lifecycle: aithernet mission submit "<prompt>" (create only), aithernet mission start, aithernet mission watch (read-only), aithernet mission retry.
  • Run in the foreground for a quick interactive session: aithernet start (Ctrl-C stops it). The managed service started by setup is the supported way to keep a node up across logout/reboot.
  • Hosted/fleet enrollment and hardware-profile installation apply to hosted or physical-SDR deployments, not the standalone software-only workstation journey.
The RF backend is a pinned, signed GNU Radio MCP component installed and verified by aithernet setup into a managed location — never a developer checkout or a manual git clone. See RF component & source notices.

SDR peer transport (1.0.0-beta.3)

  • Simulated + physical SDR peer transport. Canonical peer messages travel through authenticated encryption, fragmentation, CRC, FEC, BPSK/QPSK modulation, a channel, demodulation and reassembly to the receiving node's canonical ingress — over simulated_rf (software) or rf_ota (real SDR). The receiving node's MissionEngine policy stays authoritative; RF is never silently downgraded to IP.
  • Physical SDR transmission is available when the LOCAL operator enables and configures it on supported TX-capable hardware. The operator selects the physical parameters and is responsible for operating the radio in accordance with the rules applicable at the operating location. No vendor approval, cloud authorization, subscription, or external service is required — enable it locally with aithernet rf tx enable.
  • Aithernet validates device capabilities, enforces device + operator ranges, authenticates peers, bounds execution, and records the exact transmission plan. It never automatically chooses an operating frequency.
  • Qualification states are descriptive: software-qualified (passed in simulation/ loopback), operator-custom, and hardware-tested (confirmed by the operator on their hardware). Untested-on-your-hardware is not "certified," but the feature is never hidden. The optional live coordinator→coding-agent modem-generation test uses your own coordinator credential.

Account-bound fleet, tenant-isolated meshes & the OTA link protocol (1.0.0-beta.3)

  • Account vs node identity. Your account (email + immutable account ID) is separate from each node's Ed25519 identity. The email is never a node key and is never sent over the RF peer protocol. Packages are generic and byte-identical for every customer; the one-time enrollment code binds a node to your tenant at install time.
  • Tenant-isolated meshes. A mesh is the explicit trust group that authorizes peer communication. Two nodes that can merely reach each other — even mutually key-trusted — cannot exchange messages until they share a mesh. Cross-tenant communication requires an explicit shared mesh; it never happens just because two nodes can hear each other over RF. Meshes work fully offline in standalone mode and synchronize through the control plane when hosted.
  • Complete OTA link protocol. RTS/CTS reservation, carrier sense, randomized backoff and hidden-node mitigation on a shared channel, or a reserved TDD slot point-to-point; selective retransmission via a BLOCK_ACK bitmap; the frame set BEACON/RTS/CTS/DATA/BLOCK_ACK/ NACK/CANCEL/FIN/DELIVERY_RECEIPT. The four acknowledgement layers — link ACK, delivery receipt, mission acknowledgement and mission result — stay distinct; a radio frame ACK is never reported as mission success.
  • Fleet visibility. Signed heartbeats drive explicit online/stale/offline/degraded/never_seen/revoked states in the portal; node rename, enrollment code revocation and mesh management are in the authenticated portal.

Receive-first & permissions

  • The node is receive-first; transmission is disabled by default.
  • Non-root SDR access uses udev + the plugdev/dialout groups; the installer configures these.
  • The node makes only outbound connections to the control plane. It opens no inbound ports and never exposes its local API to the Internet.
